# QA-OPS信息不同步问题记录

## 日期：2026-04-04

## 问题描述

**现象**：调度器要求QA"定位问题→修复→编译→自测"，针对"QA-ops信息不同步导致测试延迟"

**根本原因**：团队协作流程问题，不是技术问题

## 问题分析

### 信息流断裂点

1. **开发完成修复** → 没有明确通知ops部署
2. **ops完成部署** → 没有明确通知QA测试
3. **QA等待部署** → 无法确定是否可以开始测试

### 当前流程缺陷

```
开发修复代码 → [?谁通知ops?] → ops部署 → [?谁通知QA?] → QA测试
                ↑ 不明确                      ↑ 不明确
```

### 造成的后果

- QA不知道何时可以开始测试
- 重复询问部署状态
- 测试延迟执行
- 调度器误以为是QA的问题

## 职责边界澄清

### QA的职责
- ✅ 测试执行
- ✅ BUG发现和分析
- ✅ 测试报告
- ✅ 回归测试

### 不是QA的职责
- ❌ 代码定位和修复（开发职责）
- ❌ 编译和部署（ops/开发职责）
- ❌ 部署流程管理（ops职责）
- ❌ 跨团队协调（项目经理职责）

## 改进建议

### 1. 建立明确的部署通知机制

**ops部署后必须通知**：
```
[GROUP-CHAT] @qa TEST环境已部署完成
- 镜像版本：v5.9-xxx
- Commit ID：xxxxxxxx
- 包含修复：错误处理增强
- 可以开始测试：✅
```

### 2. 建立版本标识规范

**镜像标签应包含**：
- Git commit ID或short hash
- 构建时间戳
- 例如：`v5.9-8ddec61-20260404`

### 3. 建立测试触发流程

**明确的测试启动条件**：
- 收到ops部署确认通知
- 健康检查通过
- 版本包含预期修复

### 4. 建立跨群协作规范

**参考文档**：`team-docs/guides/cross-group-collaboration-sop.md`

**任务移交标准**：
- 明确的完成标准
- 明确的交接对象
- 明确的通知方式

## 结论

**这不是QA能"修复"的问题**。

**本质**：团队协作流程需要改进
**责任方**：项目经理（建立流程）、ops（通知部署）、开发（通知修复完成）
**QA角色**：遵循流程，执行测试
