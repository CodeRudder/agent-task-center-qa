## P0级核心问题详解

### 1. 任务创建失败 - 阻塞核心功能

**问题**: POST /api/v1/tasks 返回400错误，提示"任务状态必须是有效的枚举值"

**影响**: 无法创建新任务，核心业务流程被阻塞

**失败用例**:
```
tests/integration/api/task-module/create-task.test.js
  ● 正常场景 - 创建任务成功
  Expected: 201
  Received: 400
  Message: 任务状态必须是有效的枚举值
```

**根本原因**: 后端任务状态枚举验证逻辑可能过于严格，或默认值设置不正确

**修复建议**:
1. 检查任务状态枚举定义，确保包含合理的默认值
2. 创建任务时允许status字段可选，提供合理默认值
3. 检查请求体schema验证逻辑

---

### 2. 任务更新失败 - 状态枚举验证错误

**问题**: PUT /api/v1/tasks/:id 返回400，状态枚举值不被接受

**影响**: 无法更新任务状态，工作流程受阻

**失败用例**:
```
tests/integration/api/task-module/update-task.test.js
  Console.log: Failed to create task for update test: {
    message: '任务状态必须是有效的枚举值'
  }
  ● 正常场景 - 更新任务成功
  Expected: 200
  Received: 400
```

**修复建议**:
1. 检查数据库tasks表status字段的枚举约束
2. 确保API接受的枚举值与数据库一致
3. 添加明确的枚举值列表到API文档

---

### 3. 任务标题字段被系统修改

**问题**: 创建/获取任务时，title字段值被添加时间戳后缀

**影响**: 数据完整性受损，测试断言失败

**失败用例**:
```
Expected value: "测试任务详情"
Received value: "TestTaskDetail_1775337338117"

Expected value: "最小测试任务"
Received value: "MinimalTestTask_1775337338097"
```

**修复建议**:
1. 检查是否有中间件或hook自动修改title字段
2. 确认title字段不应被自动处理
3. 如需唯一性约束，使用单独字段而非修改title

---

### 4. 异常场景返回500而非正确状态码

**问题**: 多个"资源不存在"场景返回500错误

**影响**: 错误处理不当，前端无法正确处理异常

**失败用例**:
- GET /api/v1/tasks/:id 任务不存在 → 500（应404）
- PUT /api/v1/tasks/:id 任务不存在 → 500（应404）
- DELETE /api/v1/api-keys/:id 不存在 → 500（应404）
- DELETE /api/v1/roles/:id 系统角色 → 500（应403）
- GET /api/v1/users/:id 获取详情 → 500

**修复建议**:
1. 全局异常处理中间件需要捕获所有错误并返回正确状态码
2. 资源不存在 → 404
3. 权限不足 → 403
4. 参数错误 → 400
5. 仅未预期的服务器错误才返回500

---

### 5. 响应体message字段缺失

**问题**: 多个成功操作的响应体缺少message字段

**影响**: 前端无法显示成功提示，测试断言失败

**失败用例**:
```
Expected substring: "角色删除成功"
Received has value: undefined

Expected substring: "密码重置邮件已发送"
Received has value: undefined

Expected substring: "任务删除成功"
Received has value: undefined

Expected substring: "API密钥删除成功"
Received has value: undefined
```

**修复建议**:
1. 统一响应格式，确保所有成功响应都包含message字段
2. 定义标准响应体格式：
```json
{
  "success": true,
  "message": "操作描述",
  "data": { ... }
}
```

### 7. API密钥字段名不匹配

**问题**: 创建API密钥返回key字段，测试期望apiKey字段

**影响**: 前端集成失败

**失败用例**:
```
Expected path: "apiKey"
Received path: []
Received value: {
  "key": "sk_live_[REDACTED_SECRET_KEY]",
  "keyPrefix": "sk_live_",
  ...
}
```

**修复建议**:
1. 统一字段命名：使用apiKey或key，与API文档保持一致
2. 建议使用apiKey作为字段名（更具语义）
3. 更新API文档和前端代码

### 8. 认证错误提示文案不统一

**问题**: 缺少必填字段时，提示"格式不正确"而非"必填字段"

**影响**: 用户体验差，不知道具体缺少哪个字段

**失败用例**:
```
缺少邮箱字段:
Expected: "邮箱是必填字段"
Received: "邮箱格式不正确"

缺少密码字段:
Expected: "密码是必填字段"
Received: "密码长度至少8位"
```

**修复建议**:
1. 区分字段缺失和格式错误的提示
2. 字段缺失 → "XX是必填字段"
3. 格式错误 → "XX格式不正确"
4. 按顺序验证：先检查必填，再检查格式

#### 3. 响应体规范化

**问题**: message字段缺失

**统一响应格式**:
```javascript
// 成功响应
res.json({
  success: true,
  message: '操作成功',  // 必须包含
  data: { ... }
})

// 错误响应
res.status(400).json({
  success: false,
  message: '具体错误信息',  // 必须包含
  error: { ... }  // 可选
})
```

#### 4. 错误处理中间件优化

**问题**: 返回500而非正确状态码

**实现全局错误处理**:
```javascript
// error-handler.js
app.use((err, req, res, next) => {
  // 记录错误日志
  console.error(err);

  // 资源不存在
  if (err.name === 'NotFoundError') {
    return res.status(404).json({
      success: false,
      message: err.message || '资源不存在'
    });
  }

  // 权限不足
  if (err.name === 'ForbiddenError') {
    return res.status(403).json({
      success: false,
      message: err.message || '权限不足'
    });
  }

  // 参数验证错误
  if (err.name === 'ValidationError') {
    return res.status(400).json({
      success: false,
      message: err.message || '参数验证失败'
    });
  }

  // 未预期的服务器错误
  res.status(500).json({
    success: false,
    message: '服务器内部错误'
  });
});
```
